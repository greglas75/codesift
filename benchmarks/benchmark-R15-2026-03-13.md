## Root Cause: Budget Truncation

**Problem:** Agent used `token_budget: 12000` per call with 4-5 sub-queries → ~2,400 tokens per query → last 1-2 queries truncated.

Token budget needed per call (without truncation):
- 4 sub-queries × ~5,700 tokens/query = **~22,800 tokens minimum**
- Agent used 12,000 → last query consistently truncated

**Accumulation paradox:** Fewer calls but bigger results = context grows faster.
- 4 calls × 12,000 result tokens = 48,000 tokens in context
- Each subsequent call re-receives all previous results
- Call 4 input ≈ 60,000+ tokens just from accumulated results

Compare Auggie: results per call likely ~5,000 tokens (not 12,000) → accumulates to ~20,000 by call 4

---

## Why More Tokens Than R14 (Despite Fewer Calls)

R14: 22 calls × ~500-1000 tokens per BM25 result = slow accumulation, small context per call
R15: 4 calls × 12,000 tokens per semantic result = fast accumulation, large context per call

**Total context tokens (cumulative input):**
- R14: small results, 22 calls → moderate accumulation
- R15: 12K results, 4 calls → 48K accumulated in final turn

## Root Problem vs Auggie

Auggie returns ~5,000 tokens per codebase_retrieval call (not 12,000). Their retrieval engine:
1. Returns fewer but more precisely relevant results
2. Deduplicates aggressively across the whole response
3. Probably uses RRF or learned ranking, not pure cosine similarity

Our semantic returns 11 files because top_k=15 chunks happen to span 11 files — some are test files that are only tangentially related.

---

## Fix Applied (R16)

1. **Chunk merge fix**: overlapping chunks from same file now merged (no duplicate lines) — reduces per-query token count
2. **Need**: lower top_k (7-8) to reduce result size per query
3. **Need**: higher budget per call (20,000-25,000) to prevent truncation
4. **OR**: reduce sub-queries per call (2-3 per call, 6-8 total calls)
