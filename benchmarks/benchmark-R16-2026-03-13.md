# Benchmark R16: CodeSift Semantic + RRF Auto-Decomposition

**Date:** 2026-03-13
**Project:** promptvault (Next.js 14+, TypeScript, Prisma)
**Model:** Claude Sonnet 4.6
**Index:** CodeSift 15,695 symbols / 5,622 chunks (OpenAI text-embedding-3-small, 1536 dims)
**Format:** Plain text (Path: file.ts\n   46\t[code]) — Auggie-style

---

## Changes from R15

1. **RRF auto-decomposition**: queries >8 words split at natural connectors (and/or/from/to/with/using/for/via/then) into 2 sub-queries. Each sub-query ranked independently, results merged via RRF: `score(d) = Σ 1/(60 + rank_i(d))`. Single OpenAI batch call for both embeddings.
2. **Agent strategy**: 4 codebase_retrieval calls × 3 semantic sub-queries each, top_k=5, token_budget=20000 (vs R15: 4 calls × 4-5 sub-queries, token_budget=12000)

## Why R16 Dramatically Better Than R15

| Factor | R15 | R16 |
|--------|-----|-----|
| top_k per query | 15 | 5 |
| token_budget per call | 12,000 | 20,000 |
| Result tokens per call | ~12,000 (at limit) | ~5,800 (avg) |
| Budget truncation | YES (T10 got 5/10) | NO |
| Context accumulation | 48K accumulated by call 4 | ~23K accumulated by call 4 |

**Key insight**: smaller top_k (5 vs 15) × same 4 calls = 60% less result data per call → 60% less context accumulation = much lower total tokens. The token budget wasn't the bottleneck; result size was.

## Quality Issues to Fix (R17)

### T1 — Wrong function retrieved
Query: `"createRisk function definition params return"`
Retrieved: `createRisksFromAnalysis` (bulk create, NOT `createRisk`)
Fix: More specific query — `"createRisk single risk creation documentId workspaceId"`

### T7 — ENTITY_NOT_FOUND not found
Query: `"ENTITY_NOT_FOUND error code definition throw"`
Retrieved: other error codes
Fix: Use `{"type":"text","query":"ENTITY_NOT_FOUND"}` instead of semantic for exact string lookup

### T2 — Partial importers
Fix: Use `{"type":"text","query":"from.*risk.service","regex":true}` alongside semantic for exhaustive import search

### T9 — Only 3 of 15 $transaction usages
Fix: Use `{"type":"text","query":"prisma.\\$transaction"}` for exhaustive coverage

**Pattern**: semantic is great for conceptual/architectural questions (T4, T6, T10), but BM25/text is better for exact-string lookups (T1 function name, T7 error code, T9 specific API). R17 should mix both strategies.
