## Critical Finding: Tool Call Overhead is Negligible

R23 had **4 tool calls** vs R18's **11 tool calls** — yet total tokens were nearly identical (69,161 vs 69,179 = **18 tokens difference**).

This empirically proves that **tool call overhead is ~10-100 tokens per call**, NOT 500-800 as previously estimated. The main token cost drivers are:

| Driver | Est. tokens | Source |
|--------|-------------|--------|
| System prompt + conversation structure | ~45,000 | Fixed |
| Result data from codebase_retrieval | ~19,500 | Data |
| Model answers + reasoning | ~4,500 | Variable |
| **Total** | **~69,000** | |

## Hybrid Search (New in R23 codebase)

Also implemented in this session: `"hybrid"` sub-query type that:
1. Runs semantic embedding search (RRF over decomposed sub-queries)
2. Runs text/BM25 search in parallel
3. RRF-merges both ranked chunk lists
4. Returns top_k chunks from merged result

Hybrid tested on `createRisk` query:
- Text: 1,069 tokens — found call-sites first, NOT definition
- Semantic: 1,015 tokens — found definition (2nd place)
- Hybrid: 1,198 tokens — found definition (3rd place, slightly MORE tokens)

**Conclusion:** Hybrid is NOT always better. For exact function definition lookups, the text component adds noise by surfacing call-sites, pushing the definition down in rank. Manually-selected semantic/text per query (R18 approach) remains optimal.

---

## Why Form-Style Answers Don't Save Tokens

R23 (form answers) ≈ R18 (prose answers) in total tokens. The form format saved ~2K in answer verbosity but this was offset by:
- More context accumulation from 3 calls vs 2 calls
- 3 calls: 7K + 7K + 7K = 21K accumulation overhead
- 2 calls: 6K accumulation overhead
- Extra accumulation: 21K - 6K = 15K MORE with 3 calls

The form-style answers are useful for **reducing Bash tool calls** (agent parses structured output more easily) but don't reduce total token count vs 2-call with prose answers.

## Remaining Gap Analysis (Updated)

All prompt engineering approaches cluster at ~69K. The 5K gap to Auggie appears to be:

| Source | Estimate | Confidence |
|--------|----------|-----------|
| Fixed context overhead difference | ~3,000 | Medium |
| Model verbosity difference (run-to-run variance) | ~2,000 | Low |
| **Total** | **~5,000** | |

The **tool call count** is NOT the main driver. The gap is from:
1. Auggie's session had different system context (CLAUDE.md overhead varies by project)
2. Model naturally wrote terser answers on that specific R10 run (variance)
3. Possibly Auggie's result format is more compact (unmeasured)

## Path Forward: Server-Side Optimizations

Prompt engineering has reached its limit. To close the gap, server-side changes:

| Change | Est. saving | Risk |
|--------|-------------|------|
| Shorter line number format (`18\t` vs `    18\t`) | ~1,000 tokens | None |
| Remove "The following code sections were retrieved:" header | ~50 tokens | None |
| Remove "..." chunk separators | ~100 tokens | Minor |
| Cross-call result deduplication (already in codebase) | Already done | — |
| **Total structural savings** | **~1,150 tokens** | Low |

Even with these, realistic minimum: **~68K tokens** (still 4K above Auggie).

**Conclusion:** CodeSift R18 (#2 all-time, 69,179 tokens, 9.1/10 quality) is the realistic optimum for the current architecture. Beating Auggie's R10 64,148 appears to require either: (a) different model verbosity on a lucky run, or (b) substantially reducing result data (~15K instead of ~20K while maintaining quality).
