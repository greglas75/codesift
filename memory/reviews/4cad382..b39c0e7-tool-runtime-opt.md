## FINDINGS — MUST-FIX (all FIXED in b39c0e7)

**R-1 [MUST-FIX] Unbounded synchronous `git` on the tool hot path — blocks the whole event loop; a dirty tree can report clean**
File: `src/register-tools/runtime.ts:172`
Evidence: `execFileSync("git", …)` for `rev-parse HEAD` and `status --porcelain` with no `timeout` and no `maxBuffer`. `execFileSync` blocks the entire Node event loop; a stalled git (network FS, contended `index.lock`) hangs the MCP server indefinitely. `maxBuffer` defaults to 1 MiB → many untracked files → ENOBUFS → caught → `dirty=false`, i.e. **a dirty tree is reported clean → stale cache**. `git status` is repo-wide (a monorepo subdir triggers a full worktree scan). The repo already had `src/utils/git-head.ts:13` with `timeout: 500` for exactly this reason.
Fix applied: both spawns replaced with `statSync` on the repo git dir (`HEAD`, `index`, `logs/HEAD`), incl. resolving the `.git`-FILE (worktree/submodule) form. Zero subprocesses.

**R-2 [MUST-FIX] Memoized version token delayed index invalidation — agents got PRE-EDIT analysis**
File: `src/register-tools/runtime.ts:246`
Evidence: the whole `index|git` token was TTL-memoized (2s). `index_file` runs on a post-edit hook, so an agent that edits a file and immediately calls a cacheable tool (`analyze_complexity`, `audit_scan`, …) received the pre-edit result. Cross-model adversarial CRITICAL.
Fix applied: TTL memo **deleted**. The token is computed fresh per call (4 × `statSync` ≈ 0.3 ms) — invalidation is now instant. Possible only because R-1 removed the expensive git spawns.

**R-3 [MUST-FIX] `CODESIFT_TOOL_TIMEOUT_MS` unclamped → total outage from a config typo**
File: `src/register-tools/runtime.ts:269`
Evidence: a huge value overflows `setTimeout`'s 32-bit int (`TimeoutOverflowWarning`, reproduced on Node v24) → the timer fires after ~1 ms → **every non-exempt tool instantly returns `timed_out`**.
Fix applied: `MAX_TOOL_TIMEOUT_MS = 600_000`, clamped on both the env branch and the per-tool `timeoutMs` branch.

**R-4 [MUST-FIX] An unknown version token meant "cache forever"**
File: `src/register-tools/runtime.ts:291`
Evidence: when the token degraded to `""`, `cacheKeyFor` produced a key with no version component; `withCache` has no TTL, so the entry lived for the **entire process, never invalidating**. `src/register-tool-groups/shared.ts:95` documented the opposite guarantee.
Fix applied: `cacheKeyFor` returns `null` on a degraded token → the cache is bypassed entirely for that call. Doc corrected to state the degraded contract.

**R-5 [MUST-FIX] `index_repo` was not timeout-exempt — a working tool now always reports failure**
File: `src/register-tools/runtime.ts:59`
Evidence: `TIMEOUT_EXEMPT_TOOLS` listed `index_folder`/`index_file` but not `index_repo` (clones + indexes a remote repo — routinely >90 s). It timed out and returned `isError`, while the clone kept running and succeeded. Two entries in the set (`"index-conversations"`, `"serve"`) were not real tool names — evidence the list was written from memory, which is how `index_repo` slipped through.
Fix applied: the exempt set is now **derived from `INDEX_MUTATING_TOOLS`** (so it cannot drift), dead names dropped, and a test asserts every exempt name is a registered tool.

**R-6 [MUST-FIX] A hung handler permanently bricked its cache key**
File: `src/register-tool-groups/handler-wrappers.ts:82`
Evidence: `withCache` inserts the pending promise and removes it only on settle. A handler that never settles leaves it forever → every later identical call joins that promise, waits the full timeout, returns `timed_out` → the tool is dead for that `(repo, args)` for the daemon's life. Pre-change, each call at least re-executed.
Fix applied: eviction hook wired from the timeout branch. **Second-order bug found while fixing:** `wrapTool`'s `resolveRepo` *mutates* the args object (fills in `repo` when omitted — the common case), and the key is content-derived, so evicting with the post-call object computed a *different* key and silently missed. The bind site now snapshots args **before** the call; a mutation test pins it.

**R-7 [MUST-FIX] Cache hits were invisible to telemetry — the optimization corrupted its own measurement feed**
File: `src/register-tools/runtime.ts:349`
Evidence: on a `withCache` hit the base (`wrapTool`) is never invoked, so `trackSequentialCalls` / `recordCacheHit` / `scheduleSidecarFlush` / `trackToolCall` never fire. Every repeat call to the 8 cacheable tools vanished from `usage_stats` / `usage_hotspots` / session snapshot — the exact feed whose numbers (605 calls / 909 K tok) justified this change.
Fix applied: hits record via the same helpers the inner cache used, plus `trackToolCall`; the served response carries `⚡ cached` (appended to a copy, never mutating the memoized object).

**R-8 [MUST-FIX] `find_references`: `max_refs` unvalidated, and the batch path silently ignored the cap**
File: `src/register-tool-groups/core.ts:439`
Evidence: `max_refs: -1` → `refs.slice(0, -1)` silently **drops the last reference** and prints `+${len+1} more`; `2.5` prints `+7.5 more`. Separately, the `symbol_names` batch path returned before any cap ran — while the new description claimed "Capped at max_refs (default 50)". The batch path fans out over many symbols and is by construction the larger token sink the change targets.
Fix applied: `normalizeMaxRefs = Math.max(0, Math.floor(n))`; the batch path caps per symbol; description corrected.

**R-9 [MUST-FIX / test] The whole-feature smoke pinned the broken configuration**
File: `tests/integration/tool-wrappers-smoke.test.ts:101`
Evidence: SMOKE2 used `repo: "smoke/unregistered-repo"` so the version token was the constant `"|"` — i.e. it exercised the cache in exactly the degenerate state where invalidation was broken, and the correct fix (R-4) would have made it fail. A test that locks in the bug is worse than no test.
Fix applied: rewired to a real indexed temp-repo fixture; asserts exactly-once across two identical calls, `⚡ cached` on the second, and a **miss after the index changes**.
